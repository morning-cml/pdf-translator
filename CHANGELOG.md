## [未发布] · 稳健性修复：取消 / 缓存 / API 预检 / 原子写出

**一句话**：把翻译流程里几处"卡住 / 搞坏 / 白跑"的坑一次补齐——取消能立刻停、
缓存坏了能自愈、翻译前先验通、产物永不半截损坏。均带回归测试（共 **194 项**全过）。

> 尚未打 tag 发布。发版时把本节标题改为 `[1.0.2]`、`src/version.py` 升到 1.0.2、
> 推 `v1.0.2` tag 触发 CI 构建即可。

### 1. 取消立即生效（原来"按了没反应"）

`translator.translate_texts` 旧实现把所有批一次性提交线程池，`with` 退出即
`shutdown(wait=True)`——取消要等全部在途 + 排队请求跑完才停（并发 32 时很久）。
改为：队列里尚未启动的批检测到取消即原样返回、不再发请求；收结果循环即时
`break`；executor 改 `shutdown(wait=False, cancel_futures=True)` 不等在途。前端
点取消即时禁用按钮 + "正在取消…"，不再像没反应。实测整篇 1.5s 的活取消后 0.3s 返回。

### 2. 所有产物原子写出 + 精准清理临时残留

PDF / Word / PPT / 文本原来直接写最终路径，中途失败或被取消会在目标处留下
打不开的半截文件（也是"重译同一篇却生成不出来"的元凶之一）。改为先写 `.part`、
成功才 `os.replace` 落位；失败即删、绝不留半截；最终文件被占用（正被 PDF 阅读器
打开）时自动换 `X (1).pdf` 保住成果并回传实际路径。启动时 `sweep_temp`、缓存加载
时删残留 `.tmp`——只删本程序约定命名（`.tmp`/`.part`）的文件，绝不递归/通配。

### 3. 翻译前必做 API 联通性预检

原来只有手动"测试连接"按钮，正式翻译前什么都不查——Key 错 / 连不上会让每一批都
失败、白跑一整篇还产出一份未翻译的文档。现 webui 每个任务、CLI 每次运行翻译前
**强制预检**（离线 mock 跳过），失败点名具体服务（DeepSeek / Kimi / 智谱 / 豆包 /
OpenAI / Ollama 本地）并中止、不产出文件。

### 4. 缓存自愈 + 忽略缓存重译

- **坏缓存自愈**：读缓存时校验公式占位符，命中值若占位符与原文对不上（损坏 /
  旧版本写入 / 磁盘半写）一律作废重译，不再拿坏译文贴出错位公式。
- **忽略缓存重译**：新增 web 高级选项勾选框、CLI `--refresh-cache`——无视旧缓存
  重译、并用新结果覆盖旧的坏缓存（一次性动作，不落盘持久化）。

### 5. 测试隔离（工程健康）

用例原来会读开发机上真实的 `config.json`，本机存过的 `output_mode=sidebyside`
等设置渗进断言，造成"CI 干净全绿、本机却 6 项红"的名实不符。`conftest` 增加
session 级隔离夹具，把 `CONFIG_PATH` 指到空临时目录、清掉 `DEEPSEEK_*` 环境变量，
用例一律从纯默认配置起步。

对应提交：`95cbf57`(取消) `8a3023d`(原子写出) `e74c453`(API 预检)
`9b41a08`(缓存自愈) `93fb0f2`(测试隔离)

## [1.0.1] — 2026-07-20 · 首个公开发布版

**一句话**：修掉一个"看不见的"严重缺陷（完整版的 OCR 其实没打进包），
把构建产物管起来（只归档不删除），并**将仓库转为公开**、补上门面 README。

**这是第一个真正对外分发的版本**——v1.0.0 虽已完成全部功能，但从未打 tag、
从未发布，其构建产物还带着下面这个 OCR 缺陷。

### 1. 修复：完整版（full）实际不具备 OCR 能力 ⚠️

打包产物里没有 `rapidocr_onnxruntime` 包，也没有它的三个 `.onnx` 模型。但它
**表现得毫无异常**：exe 正常启动、界面正常、体积还比 lite 大 150MB
（`onnxruntime` 与 `cv2` 是被 `layout_model.py` 的函数内 import 捎带进去的），
所以从构建日志和体积上完全看不出问题。实际后果：full 与 lite 对扫描版 PDF
表现一模一样，那 150MB 白占。

- **根因**：`src/ocr.py` 里 rapidocr 是函数内延迟 import，PyInstaller 静态分析
  扫不到；且其模型与 `config.yaml` 是**包数据**，即便加 `--hidden-import` 也不会
  被带上，必须 `--collect-all`。
- **修复**：full 档增加 `--collect-all rapidocr_onnxruntime`。
- **防复发**：新增 `verify_build()` 打包后自检——full 必须含 rapidocr 包与 ≥2 个
  模型、lite 必须不含，两档都验 exe 与前端资源，不达标直接判构建失败。这类
  "看着对、实则少功能"的缺陷只能靠打包后验产物内容发现。
- 顺带修：缺 OCR 时的提示分场景。打包版用户没有 pip，原来那句"请运行
  `pip install rapidocr-onnxruntime`"对他们是死路，改为提示换用完整版。

### 2. 仓库转公开 + 根目录 README

起因是要把 exe 发给别人：AGPL 要求接收者能拿到对应源码，而包内《首次使用
请看我.txt》指向的地址必须真的打得开，否则分发即不合规。

- 新增**根目录 `README.md`**：此前只有 `程序/README.md`（面向开发者讲架构），
  GitHub 首页是空白的。新 README 面向使用者，含下载入口、full/lite 怎么选、
  "整个文件夹一起用、只拷 exe 跑不起来"的提醒、能力清单与文档地图。
- 转公开前做了安全检查：`config.json`（含真实 Key）全历史从未入库，全历史
  扫描无 `sk-` 形态密钥、无硬编码 api_key/token/password。
- 更新 GitHub 仓库描述（原描述只提"英文文档（PDF/Word）翻译成中文"，已不覆盖
  PPT/Markdown/字幕与多语言方向）并补齐 14 个 topics。

### 3. 构建产物保全

**动机**：v1.0.0 的 exe 产物在本机丢失。查因发现两处：`release/` 被 `.gitignore`
整体排除（git 无法找回），且 `build.py` 每次构建都会 `shutil.rmtree()` 同版本同
profile 的旧目录——**重建一次，上一次的产物就没了**，且无任何提示。

**改动**：
- **构建脚本不再删除任何历史产物**。目标目录已存在时**默认中止**并提示改版本号；
  确要重建同一版本需显式 `--overwrite`，旧产物被**移动**到
  `release/_history/<名字>-<时间戳>/` 留档而非删除。`--clean` 也只动 `build/`。
- **新增版本台账 `release/RELEASES.md`**，由构建脚本自动维护并**纳入 git**
  （`.gitignore` 改 `release/*` 并对其单独取反——用 `release/` 会整体排除目录、
  取反不生效）。二进制仍不入库（走 GitHub Releases），但"发过哪些版本、何时、
  由哪个提交构建、多大"永久可查，本地产物丢了也能从对应提交精确重建。
- `--list` 改为同时列出当前版本与 `_history/` 中被顶替的产物，并显示来源提交。

**产物**：full 314.8 MB（rapidocr 15.5 MB 完整入包，det/rec/cls 三个模型齐全）、
lite 149.6 MB（确认不含 OCR）。两版均实测启动正常并返回 HTTP 200，自检通过。
pytest 179 项全过。

对应提交：`ad9260b`(产物保全) `34edc51`(OCR 修复+自检) `dcd495e`(转公开+README)

## [1.0.0] — 2026-07-20 · 正式版

**一句话**：从"英译中 PDF 工具"成长为**多格式、多语言、有测试、可打包、已开源**的
完整文档翻译器。

**相比 v0.9 新增**：
- **多语言方向（B5）**：源 11 种（含自动检测）× 目标 10 种（中/英/日/韩/德/法/西/俄/
  葡/意）。此前只有英→中。含脚本感知断行（欧洲重音不断词、CJK 逐字、韩俄按词）、
  目标语感知的"是否已翻译"判定、按语言对的质量长度带、缓存键含语言对。
- **语言→模型推荐**：界面内联提示，选目标语后建议更合适的服务并可一键切换
  （中英用 DeepSeek、欧洲语言用 OpenAI…）。
- **PowerPoint（.pptx）支持（A3）**：文本框、表格、演讲者备注，样式保真。
- **精简版（lite）构建**：不含 OCR，体积约为完整版一半（149MB vs 298MB）。
- **首次运行引导**：未配置 Key 时给三选一引导页（DeepSeek 申请 / Ollama 免费离线 /
  离线试用），而非等点击后报错。
- **CI（GitHub Actions）**：推送自动跑测试；打 `v*.*.*` tag 自动构建 full+lite 并发布。
- 修复：build.py 的 full/lite 同版本号互相覆盖（改为 `release/v<版本>-<profile>/`）；
  中→英/日→中"原样返回"bug（`_worth_translating` 曾写死"已是中文就跳过"）。

**支持格式**：PDF · Word(.docx) · PowerPoint(.pptx) · Markdown · TXT · SRT
**支持服务**：DeepSeek · Kimi · 智谱GLM · 豆包 · OpenAI · Ollama本地 · 自定义
**测试**：179 项 pytest（CI 干净环境 168 过 + 11 跳过依赖真实论文的用例）

对应提交：`2ece5d7`(lite+引导) `732fb85`(CI) `72ac8f5`(多语言) `a063062`(PPTX)

## [0.8] — 2026-07-19 · 核心引擎（初始提交 `3b9aeed`，含 8 轮 pre-git 迭代）

首次入库时已具备的能力（此前 8 轮迭代累积，细节见交接说明）：

1. **精确回填内核**：PyMuPDF 精确抹除原文字符（深色背景零露白）+ reportlab 覆盖兜底，
   自动选择与回退。
2. **行内公式保护**：检测 → `⟦Fn⟧` 占位 → 译后校验 → 矢量回贴。
3. **共享重排引擎**：中文断行禁则、逐行避障绕图、向下扩展、自适应缩号、两端对齐。
4. **结构还原**：数据驱动分栏、标题层级（加粗+原色）、参考文献免译保排、跨栏断句缝合。
5. **扫描版 OCR**：RapidOCR 本地离线，无需显卡。
6. **可选版面分析模型**：DocLayout-YOLO（识别表格/图区/独立公式）。
7. **翻译侧**：六家服务预设、术语库、全文语境注入、持久化缓存、失败降级、成本预估、
   试译模式（只翻前 N 页）。
8. **三种输出**：纯中文 / 双语·前后页 / 双语·左右对照宽页。
9. **网页皮肤界面**：分层前端架构，皮肤=一个 CSS 文件（含海绵泡泡、清爽白两套）。
   tkinter 版冻结为备用。

> 这 9 项对应交接说明里的「第一~八轮迭代」，是本项目的技术底座。
