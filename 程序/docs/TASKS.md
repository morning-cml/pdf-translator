# 优化任务清单（第七轮，2026-07-18 制定）

> 来源：对照 PDFMathTranslate（EMNLP 2025）、BabelDOC/沉浸式翻译、有道/知云等
> 主流方案与本项目代码复审得出的 12 项改进。按序实现，每项完成后跑回归。
> 状态标记：[ ] 待做 · [x] 完成 · [~] 部分完成/可选启用

## 第一梯队：成本与可靠性（收益最直接）

### T1 [x] 持久化翻译缓存
- **动机**：现缓存仅在内存（`BaseTranslator._cache`），程序一关即失；同一篇
  论文每次重跑都全价耗 token。PDFMathTranslate 用磁盘缓存"防止冗余计算"。
- **实现**：新增 `src/transcache.py`。键 = sha1(模型|领域/上下文|原文)，值 =
  译文；存 `程序/cache/translations.json`（.gitignore 排除），原子写入，
  每批完成即落盘。Mock 不写缓存；占位符校验失败回退原文的段**不缓存**
  （下次重跑应再试）。CLI `--no-cache` / config `use_cache` 可关。
- **验收**：同文档第二次翻译时日志显示"缓存命中 N 段"，仅对新增/失败段计费。

### T2 [x] 单批失败不中断（失败降级）
- **动机**：现在任何一批重试 3 次仍失败会抛异常终止整次任务，90% 进度崩溃
  = 前面 token 全白费。
- **实现**：`translate_texts` 逐 future 捕获异常；失败批内各段回退原文、
  计入 `failed_texts`，继续跑完。pipeline 结束时提示"N 段失败已保留原文，
  重跑可补齐（缓存段不重复计费）"。失败段不写缓存。
- **验收**：人为断网一批，任务仍完成并给出统计；重跑仅补失败段。

### T3 [x] 试译模式（部分页翻译）
- **动机**：商业工具皆有预览；先花几分钱看前几页效果，满意再翻全篇。
- **实现**：`translate_pdf(..., max_pages=N)`：整篇解析，仅翻译前 N 页的块，
  其余页原样保留。CLI `--pages N`；GUI"试译前 N 页（0=全部）"。
- **验收**：`--pages 2` 只对前 2 页产生译文且只为这些段计费，其余页原版。

### T4 [x] 双语左右对照输出
- **动机**：现有 bilingual 为"原文页/译文页交替"，对照要翻页；沉浸式翻译/
  BabelDOC 的核心体验是并排对照。PDF 最实用形式 = 左原右译宽页。
- **实现**：新 output_mode `sidebyside`：每页生成 2W×H 宽页，左贴原文页、
  右贴译文页（fitz `show_pdf_page`；reportlab 路径用 pypdf Transformation
  平移合成）。GUI 输出模式三选：纯中文 / 双语·前后页 / 双语·左右对照。
- **验收**：宽页 PDF 左右内容对齐、页数与原文一致。

### T5 [x] 翻译前成本预估
- **动机**：透明与信任；避免"不知道要花多少"的顾虑。
- **实现**：按待译字符估算 token（英文 ≈ chars/3.6 + 提示词开销 ~18%，输出
  ≈ 输入×1.1）；config 可填 `price_in`/`price_out`（元/百万token，默认 0）
  ——填了则同时显示金额。pipeline 在解析后、请求前上报一行预估；缓存命中
  段不计入。
- **验收**：日志出现"预计 ~N 段 / ~M tokens（约 ¥X）"。

## 第二梯队：翻译质量与观感

### T6 [x] 全文上下文注入（口径一致性）
- **动机**：各批独立翻译，术语/口径可能批间漂移；沉浸式翻译等会携带文档
  语境。
- **实现**：pipeline 取 p1 最大字号可译块（标题）+ 首个长段（摘要）前 ~350
  字符 → `translator.set_context()`；system prompt 附"本文主题背景（仅供
  理解不要输出）"。上下文哈希并入缓存键。
- **验收**：system prompt 含标题/摘要；缓存键随上下文变化。

### T7 [x] OCR 行→段合并
- **动机**：扫描版按识别行成块（p1 达 22 块 vs 文字版 10 块），句子切碎、
  翻译上下文受损。
- **实现**：`_group_blocks(from_ocr=True)` 时段落合并阈值放宽（gap_limit
  ×1.35；字号突变阈值 0.25→0.45，识别框字号本就抖动），把逐行块并成段。
- **验收**：扫描样张 p1 块数明显下降（≈10±3），分栏/绕图回归不受影响。

### T8 [x] 两端对齐（justify）
- **动机**：原文两端对齐、译文左对齐，右缘参差影响"原生感"。
- **实现**：`layout._flow` 行收尾时：非段末行、填充率 >0.82、单元数 ≥3 →
  把剩余宽度均摊到单元间距（每处上限 0.30×字号）。段末行保持左对齐。
- **验收**：正文右缘整齐；无过大字距；公式行/标题不受影响。

### T9 [x] 翻译服务预设
- **动机**：base_url 本就兼容一切 OpenAI 接口；下拉预设把"可以"变成"好用"，
  Ollama 本地 = 免费离线翻译（PDFMathTranslate 多服务的低配实现）。
- **实现**：GUI"服务"下拉：DeepSeek 官方 / OpenAI / Ollama 本地 / 自定义；
  切换时预填 base_url+模型（均可再编辑）。config 不变（仍存最终值）。
- **验收**：切换预设自动填入对应 base_url/模型；自定义可手填。

### T10 [x] 领域可配置
- **动机**：提示词与术语库偏计算机；其他学科译文口径可更准。
- **实现**：config 新增 `domain`（默认"计算机科学"）；SYSTEM_PROMPT 模板化
  `{domain}`；GUI 可编辑下拉（计算机科学/通用学术/生物医学/物理学/数学/
  电子工程）。领域并入缓存键。
- **验收**：切换领域后提示词随之变化，缓存不串。

## 第三梯队：结构性增强

### T11 [x] 版面分析模型（DocLayout-YOLO，可选插件，已启用）
- **动机**：PDFMathTranslate/BabelDOC 与本项目的本质差距——用检测模型替代
  启发式识别表格/图区/独立公式，复杂版式鲁棒性质变。
- **实现**：新增 `src/layout_model.py`：`models/doclayout_yolo_docstructbench_
  imgsz1024.onnx` 存在即自动启用（onnxruntime 随 OCR 已装，无 torch）。每页
  一次检测（CPU ~0.4s）：`table`/`isolate_formula` 区内可译块改为不译（保
  原样），`figure`/`table` 并入障碍物（补启发式盲区）。任何一次推理失败即
  整体停用（`broken` 标记），绝不影响主流程；删除模型文件即完全关闭。
- **完成情况**：模型已从 hf-mirror（wybxc/DocLayout-YOLO-DocStructBench-onnx，
  72MB）下载至 `程序/models/` 并启用。真实论文验证：p2/p3/p6/p8 图区检测
  准确、**零误报**（可译块数不变 132）；本论文无正文表格，表格保护逻辑待
  含表格文档进一步实测。
- **验收**：放入模型后表格/独立公式区不再被译文侵扰；无模型时行为与现在
  完全一致。✔

### T12 [x] 跨栏段落重排（翻译单元合并）
- **动机**：左栏尾段与右栏首段常是同一句话被腰斩，分开翻译则断句处上下文
  受损（已知限制 8.7）。
- **实现**：pipeline 翻译前配对：同页"左栏最后可译块 + 右栏第一可译块"
  （及跨页"本页最后 + 次页第一"），条件 = 前块不以终止标点结尾 且 后块以
  小写字母开头。配对块合并为一个翻译单元送译；译文按源长度比例在最近的
  句读处拆回两块。仅正文块参与（标题/OCR/引用不配对）。
- **验收**：真实论文中被腰斩句（如 p14"software and | methodology…"）合并
  翻译，断句处译文连贯；拆回后两栏各自排版正常。
