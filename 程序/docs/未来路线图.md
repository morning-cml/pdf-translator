## 一、现状盘点（2026-07-19）

**代码规模**：Python 3887 行（14 个模块）+ 前端约 740 行（HTML/CSS/JS）。

**已具备的能力**（八轮迭代，均经真实运行验证）：

| 维度 | 现状 |
|---|---|
| 版式保真 | PyMuPDF 精确抹除 + 公式矢量回贴 + 共享重排（避障/禁则/两端对齐/缩号） |
| 结构理解 | 数据驱动分栏、标题层级（加粗/颜色）、参考文献免译保排、跨栏断句缝合 |
| 输入覆盖 | 文字版 PDF + 扫描版（RapidOCR 本地离线） |
| 输出形态 | 纯中文 / 双语前后页 / 双语左右对照 |
| 翻译侧 | 6 家服务预设、术语库、全文语境、占位符保护、持久缓存、失败降级、成本预估 |
| 增强插件 | DocLayout-YOLO 版面模型（可选，删文件即关） |
| 界面 | 网页版（皮肤化，2 套主题）为唯一正式界面；tkinter 版冻结备用 |

**与开源标杆的差距**（对照 PDFMathTranslate / BabelDOC 论文）：二者已转向
**中间表示（IR）架构**——把视觉版面与文本语义解耦，先在语义层翻译再重新锚定
回版面，从而支撑多行公式重建、表格结构化、术语一致性。我们仍是「块 + 坐标」
的直接操作模型，简单可靠，但在复杂表格与嵌套结构上天花板较低。

**已知短板**（详见 handoff §8）：表格内文字处理未经充分验证；旋转页退兜底
后端；扫描版倒置不纠正、深色纸背露白；数字空格不完全统一；仅支持 PDF 单一
格式；仅英译中一个方向。

## 二、⚠️ 最高优先级：商业化的许可证前置条件

**这是本次调研最重要的发现，会直接决定你能不能卖、怎么卖。**

| 组件 | 许可证 | 商用影响 |
|---|---|---|
| pdfplumber | MIT | ✅ 自由商用 |
| pypdf | BSD | ✅ 自由商用 |
| reportlab（开源版） | BSD | ✅ 自由商用 |
| RapidOCR | Apache 2.0 | ✅ 自由商用 |
| requests / onnxruntime | Apache 2.0 | ✅ 自由商用 |
| **PyMuPDF** | **AGPL-3.0** | ⚠️ **分发即须开源整个产品**，否则需向 Artifex 购买商业授权（业界报价约 $10k–50k/年） |
| **DocLayout-YOLO 模型** | **AGPL-3.0** | ⚠️ 同上，商用需 Ultralytics 企业授权 |

**关键点**：AGPL 的传染性对「通过网络提供服务」同样生效——就算你不发安装包、
只做在线 SaaS，用了 PyMuPDF 也要开源全部代码。

**三条可选路线**（建议按 A → B 评估）：

- **路线 A：开源 + 增值服务（推荐先走）**
  产品本体按 AGPL 开源，靠「云端算力/额度、皮肤与模板、企业支持、私有部署
  实施」变现。合规成本为零，且开源本身是获客渠道（PDFMathTranslate 靠这条
  路拿到 22 万+ 下载）。
- **路线 B：换掉 AGPL 组件做闭源商用**
  把 PyMuPDF 从「首选后端」降为「用户自行安装的可选增强」，闭源版默认走
  reportlab（BSD）；版面模型换 Apache/MIT 许可的替代品或自训。
  **需要的工作量**：把 reportlab 后端提升到接近 fitz 的质量（见方向二 D3）。
  这是把「技术债」变成「商业选项」的关键投资。
- **路线 C：购买商业授权**——单人/早期项目不现实，规模化后再评估。

**立即可做的低成本动作**：在仓库补 `LICENSE` 与 `第三方许可.md`，声明各依赖
许可与当前分发方式；决定路线前，**不要以闭源形式对外分发含 PyMuPDF 的版本**。

## 三、五个改进大方向

### 方向一：格式覆盖面 —— 从「PDF 工具」到「文档翻译工具」

竞品（沉浸式翻译、Doclingo、OpenL、Belindoc）普遍已覆盖 PDF/DOCX/PPTX/EPUB/
XLSX/SRT。这是**用户感知最直接、技术风险最低**的扩张方向，且不受 AGPL 影响
（Office/EPUB 处理库均为宽松许可）。

- **A1 DOCX 支持**（`python-docx`，MIT）：按 run 级替换文本，天然保留样式，
  实现难度远低于 PDF。**建议作为下一个大功能的首选**。
- **A2 EPUB 支持**（`ebooklib`，AGPL⚠️ / 或直接用 zipfile+lxml 自实现，
  避免许可问题）：电子书章节 HTML 内文本节点替换。
- **A3 PPTX 支持**（`python-pptx`，MIT）：形状内文本框替换，注意自动缩放。
- **A4 Markdown / TXT / SRT**：几乎零成本，适合先上以扩充"支持格式"清单。
- **A5 批量与文件夹级任务**：整目录递归翻译 + 汇总报告。

### 方向二：翻译与版式质量 —— 守住核心竞争力

- **B1 表格结构化处理**（当前最大质量盲区）：版面模型已能框出 table 区，
  下一步用 pdfplumber 的 `extract_table()` 拿到单元格网格，**逐单元格翻译并
  按格回填**，而非整块重排。
- **B2 术语一致性增强**：先全文扫描抽取高频专名 → 生成"本文术语表"注入每批
  （BabelDOC 论文将 document-level terminology 列为关键改进点）。
- **B3 译文自检与回译** ✅：`src/quality.py` 五类本地规则——截断（长度比
  <0.12）、啰嗦（>1.6）、数字错漏、元话语前缀、重复退化；命中则带着具体问题
  定向重译一次，重译需同时通过占位符校验与自检才采纳，否则保留首译。
  **阈值全部依据 116 段真实译文语料标定**（实测长度比 0.20–0.88），真实语料
  零误报、线上 19 段实跑零额外开销；6 类坏译文全部检出。
  关键教训：数字校验必须豁免个位数并接受中文数字写法（"study 2"→"实验二"、
  "20"→"二十"），否则会把正确译文判为错漏而白白重译。
- **B4 双语行内对照**（沉浸式翻译的招牌形态）：同一页内原文段落下方紧跟译文，
  而非整页并排——对精读场景体验更好。
- **B5 多语言方向**：中译英、日/德/法 → 中。提示词与断行规则需按语种分化
  （当前 `layout.py` 的禁则表是中文专用）。
- **B6 IR 架构演进**（长期）：借鉴 BabelDOC，把 `Block` 升级为「文档中间表示」，
  彻底分离版面与语义。**收益大但改动面极大，建议在 B1–B4 沉淀后再评估**。

### 方向三：产品化与分发 —— 让非技术用户零门槛

- **C1 免安装打包**（PyInstaller / Nuitka）：目前仍要求用户自行装 Python，
  这是最大的获客漏斗流失点。打包成单个 exe 后「下载即用」。
  ⚠️ 与许可证路线绑定：打包分发 = AGPL 触发点。
- **C2 应用化窗口**（pywebview）：把现有 `web/` 原样装进原生窗口，去掉黑色
  控制台窗口与浏览器地址栏，观感即商业软件。前端零改动。
- **C3 任务队列与历史记录**：多文件排队、历史任务可回看/重开，替代当前"一次
  一个任务"的限制。
- **C4 自动更新与错误上报**：商业化必备的运维闭环。
- **C5 用量与费用面板**：累计 token/费用统计、按文档成本明细。

### 方向四：商业化能力（承接你的谋利预期）

- **D1 皮肤市场**：`themes.json` 加 `paid` 字段 + 解锁码校验（架构已预留）。
  卡通/学科主题包是低成本高感知的付费点。
- **D2 云端版**：现有 API 语义即云端草案，前端仅改 BASE_URL。按页/按量计费，
  免去用户申请 Key 的门槛——**这是最容易变现的形态**（对标沉浸式翻译月付
  ~88 元档位）。
- **D3 reportlab 后端质量攻坚**：为路线 B 铺路，同时消除"兜底后端露白/残留
  英文"的已知缺陷。**一举两得，性价比最高的技术投资之一**。
- **D4 团队/机构版**：共享术语库、统一配额、成员管理。
- **D5 Zotero / 浏览器插件集成**：嵌入科研工作流（PDFMathTranslate 已做
  Zotero 插件，验证了该入口的价值）。

### 方向五：工程健康度

- **E1 自动化测试套件**：当前靠手工跑脚本；应沉淀为 `pytest`（解析结构快照、
  pangu/缓存/配对单测、端到端 mock）。**改动内核前的安全网，越早越省事。**
- **E2 版本管理**：项目仍未初始化 git。**建议立刻 `git init`**——八轮迭代的
  演进历史目前只存在于文档里，一旦误删无从回滚。
- **E3 结构化日志与诊断包**：出问题时一键导出环境+日志，便于远程支持。
- **E4 性能优化**：大文档（100+ 页）的解析与渲染耗时、内存占用尚未测量。
