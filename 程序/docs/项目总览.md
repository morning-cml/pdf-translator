## 一、这是什么

一个**保留排版的文档翻译工具**：把英文文档译成中文，图片、公式、表格、
版式原样保留，译文落在原文对应位置。面向科研论文，非程序员双击即用。

| 项 | 现状 |
|---|---|
| 支持格式 | **PDF · Word(.docx) · Markdown · TXT · SRT** |
| 翻译服务 | DeepSeek / Kimi / 智谱 / 豆包 / OpenAI / Ollama 本地（任意 OpenAI 兼容接口） |
| 界面 | 网页版（原生应用窗口，皮肤可换）；tkinter 版冻结备用 |
| 分发 | `build.py` 一键打包 exe，免装 Python |
| 许可 | **AGPL-3.0**（开源 + 增值服务路线） |
| 仓库 | https://github.com/morning-cml/pdf-translator（私有） |
| 规模 | 源码 5380 行 · 测试 1075 行 · 前端 737 行 · 文档 1135 行 |
| 质量 | **pytest 147 项全通过**，含真实论文结构不变量 |

## 二、已完成的能力（九轮迭代）

### 版式保真（核心竞争力）
- **双回填后端**：PyMuPDF 精确抹除原文字符（深色背景零露白）+ reportlab
  覆盖兜底，自动选择与回退
- **行内公式保护**：检测 → ⟦Fn⟧ 占位 → 译后校验 → 矢量回贴
- **共享重排引擎**：中文断行禁则、逐行避障绕图、向下扩展、自适应缩号、两端对齐
- **结构还原**：数据驱动分栏、标题层级（加粗+原色）、参考文献免译保排、
  跨栏/跨页断句缝合、表格逐单元格翻译

### 输入与输出
- 扫描版 PDF 自动 OCR（RapidOCR，本地离线，无需显卡）
- 可选版面分析模型（DocLayout-YOLO，识别表格/图区/独立公式）
- 三种输出：纯中文 / 双语·前后页 / 双语·左右对照宽页

### 翻译质量与成本
- 术语库注入、全文语境注入、领域可配置
- **持久化缓存**：同文档重跑几乎零成本（实测重译整篇仅请求 2 段）
- **失败降级**：单批失败不中断任务，重跑自动补齐
- **译文自检**：截断/啰嗦/数字错漏/元话语/重复退化五类检查，命中定向重译
  （阈值按 116 段真实语料标定，零误报）
- 试译模式（只翻前 N 页）、翻译前成本预估

### 工程与产品
- 网页前端分层架构，皮肤 = 一个 CSS 文件
- 一键打包：版本管理、自动图标、SHA256 清单、full/lite 双档、签名钩子
- git 版本管理 + 9 个提交 + 私有仓库同步

## 四、未来任务（按"性价比"排序，非按方向）

### 关于版面模型的处置（2026-07-20 决定：保留现状 = 路线 C）

DocLayout-YOLO（72MB，AGPL）目前**仅存在于开发机**，用户拿不到。评估了三条路：
A 做自动下载、B 直接移除、C 保留现状。**采纳 C**——理由：① 实测在现有真实
样本上零误报但也**无可见增益**（那篇论文无表格，图区启发式已够准）；② 缺失时
静默退回启发式，核心功能（含表格逐格翻译）不受影响，已验证；③ 其价值只在
"复杂/无框线表格"这类**尚无测试样本证明的硬场景**。故不投入做自动下载，也不
急于移除；**留作开发机上的调试增强**，待出现"表格翻译确实不好"的真实案例，
再决定自动下载 / 自训小模型 / 彻底移除。（移除它还能顺带清掉一个 AGPL 依赖，
对许可更干净——列为未来若砍则一并考虑的点。）

### 第一优先：让现有能力真正可用（补最后一公里）

1. **lite 版构建与验证** + 体积优化
   实测 lite 是否真能跑（OCR 相关 import 已排除，需确认无残留引用）；
   full 版尝试剔除未用的大依赖（如 matplotlib/scipy 若被间接引入）。
3. **首次运行引导**
   没填 Key 时给出清晰的引导页（去哪申请、大概多少钱、可否免费用 Ollama），
   而不是等用户点了翻译才报错。
4. **CI（GitHub Actions）**
   推送即跑 147 项测试；打 tag 自动构建并发到 Releases。
   开源项目标配，也终结"每次手动构建"。

### 第二优先：扩大适用面（市场杠杆最大）

5. **B5 多语言方向**：中译英、日/德/法 → 中。
   需要按语种分化提示词与断行规则（`layout.py` 的禁则表目前是中文专用）。
   **这是单项投入产出比最高的市场扩展。**
6. **A3 PPTX / A2 EPUB**：分派器已就位，新增格式只需登记一处。
7. **C3 任务队列与历史**：多文件排队、历史可回看重开，替代"一次一个任务"。

### 第三优先：商业化落地（路线 A 框架内）

8. **D2 云端版**：现有 API 语义即云端草案，前端只改 BASE_URL。
   按量计费，免去用户申请 Key 的门槛——**最容易变现的形态**。
   ⚠️ AGPL 第 13 条：联网服务同样必须向使用者提供源码。
9. **D1 皮肤市场**：`themes.json` 加 `paid` 字段 + 解锁校验（架构已预留）。
10. **开源获客**：英文 README、效果对比图、演示站。
    走路线 A，开源本身就是获客渠道（PDFMathTranslate 靠此拿到 22 万+ 下载）。

### 第四优先：质量深水区

11. **B2 术语一致性**：全文扫描抽取高频专名 → 生成"本文术语表"注入每批。
12. **B4 双语行内对照**：同页原文段落下方紧跟译文（沉浸式翻译的招牌形态）。
13. **E3 诊断包 / E4 性能**：一键导出环境+日志；测 100+ 页大文档的耗时与内存。
14. **D3 reportlab 后端攻坚**：消除兜底后端"露白/残留英文"的既有缺陷。

### 长期观察

15. **B6 IR 架构**：借鉴 BabelDOC，把 `Block` 升级为文档中间表示，彻底分离
    版面与语义。收益大、改动面极大——建议在上述沉淀后再评估。
16. **D5 生态集成**：Zotero 插件、浏览器扩展。

## 五、我的建议

**下一步做 1、2、4 这三件**（模型下载 + lite 验证 + CI）。理由：它们都是
"把已经做出来的东西真正交付到用户手里"，投入小、消除的是**当前最大的名实
不符**；CI 还能让后续每次改动自动回归，长期省力。

之后如果目标是**扩大用户**，做 5（多语言）；如果目标是**尽快变现**，
做 8（云端版）。这两者选其一集中投入，比平推所有方向有效得多。
